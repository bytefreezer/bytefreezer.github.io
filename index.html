The Problem: Post-Breach Forensics

When a breach is discovered weeks later, you need to reconstruct what happened

Traditional SIEMs Are Expensive

Enterprise SIEM solutions cost $1M+ annually. Splunk charges $150/GB, Datadog has complex per-host pricing, Oracle requires multi-billion dollar contracts. These costs make comprehensive logging prohibitive.

Vendor Controls Your Data

Your security data lives in vendor infrastructure. You pay for retention, pay for queries, pay for exports. When you want to leave, migration is painful and expensive. Compliance becomes their problem, not yours.

Limited Retention Windows

Cost-prohibitive retention means you can only afford 7-30 days. When sophisticated attacks are discovered months later, the evidence is already gone. No forensics possible.

The Solution: Security DVR

Think security camera DVR, but for all your operational data

What is a Security DVR?

Your building has security cameras. They record 24/7 to a DVR. When something happens (theft, accident, dispute), you rewind the footage and see exactly what occurred. You don't watch cameras in real-time - you investigate when needed.

ByteFreezer is the same concept for your IT infrastructure. Record everything (logs, network flows, metrics) continuously. Store compressed in your own bucket. When a breach is discovered weeks later, rewind and investigate with AI-powered queries. Complete forensics from complete history.

Installation Steps

Docker Installation

# Pull the latest image
docker pull bytefreezer/proxy:latest

# Run with environment variables
docker run -d \
  --name bytefreezer-proxy \
  -p 514:514/udp \
  -p 514:514/tcp \
  -e BF_STORAGE_ENDPOINT=s3.amazonaws.com \
  -e BF_STORAGE_BUCKET=your-bucket-name \
  -e BF_STORAGE_ACCESS_KEY=your-access-key \
  -e BF_STORAGE_SECRET_KEY=your-secret-key \
  -e BF_RETENTION_DAYS=30 \
  bytefreezer/proxy:latest

Kubernetes Installation

# Add ByteFreezer Helm repository
helm repo add bytefreezer https://charts.bytefreezer.io
helm repo update

# Install with custom values
helm install bytefreezer bytefreezer/proxy \
  --set storage.endpoint=s3.amazonaws.com \
  --set storage.bucket=your-bucket-name \
  --set storage.accessKey=your-access-key \
  --set storage.secretKey=your-secret-key \
  --set retention.days=30

Example Configuration File (config.yaml)

# ByteFreezer Proxy Configuration
server:
  listen: "0.0.0.0:514"
  protocols:
    - udp
    - tcp

# S3-compatible storage
storage:
  type: s3
  endpoint: s3.amazonaws.com  # or minio.example.com
  bucket: security-logs
  region: us-east-1
  access_key: ${BF_ACCESS_KEY}
  secret_key: ${BF_SECRET_KEY}
  path_style: false  # true for MinIO

# Data processing
processing:
  compression: zstd
  format: parquet
  batch_size: 10000
  flush_interval: 60s

# Retention policy
retention:
  days: 30
  glacier_after: 7  # Move to cold storage after 7 days

# Data sources
sources:
  - name: syslog
    type: syslog
    port: 514

  - name: netflow
    type: netflow
    port: 2055

  - name: kafka
    type: kafka
    brokers:
      - kafka1.example.com:9092
    topics:
      - security-events

Troubleshooting

Problem: Proxy not receiving data

• Check firewall rules (UDP/TCP port 514)
• Verify syslog source configuration
• Run: tcpdump -i any port 514 to verify traffic
• Check logs: docker logs bytefreezer-proxy

Problem: S3 upload failures

• Verify S3 credentials and bucket permissions
• Check bucket policy allows PutObject
• For MinIO: ensure path_style: true
• Test with: aws s3 ls s3://your-bucket

Problem: High memory usage

• Reduce batch_size in config (default: 10000)
• Decrease flush_interval for faster writes
• Set resource limits in Docker/K8s
• Monitor with: docker stats bytefreezer-proxy

Problem: Data not appearing in queries

• Wait for flush_interval (default: 60s)
• Check S3 bucket for Parquet files
• Verify metadata catalog generation
• Refresh metadata: bytefreezer catalog refresh

Can I use my own AI models?

Yes. BYOA (Bring Your Own Agent) means just provide your API key (OpenAI, Claude, Gemini) or local LLM endpoint. ByteFreezer handles all integration: metadata catalog, query optimization, context injection. No custom code needed.

What retention period should I choose?

Security teams typically choose 30-90 days for incident response. Compliance requirements (SOC2, HIPAA) may need 1 year+. Start with 30 days and adjust based on actual forensic needs. Storage costs scale with retention.

What if I want to leave ByteFreezer?

Your data is in standard Parquet format in your own storage. Just stop using our service and keep querying your data with DuckDB, ClickHouse, Athena, Spark, or any analytics tool. Zero lock-in.

Can I use ByteFreezer on-premises?

How is this different from S3 directly?

Can I monetize my data?

Yes. Anonymize and package datasets, list in ByteFreezer marketplace or sell directly. Security teams sell threat intelligence, network operators sell traffic patterns, IoT companies sell sensor data.

What storage providers are supported?

Technical Questions

What data ingestion rates does ByteFreezer support?

Single proxy instance handles 100K events/sec (8.6B events/day). Horizontal scaling supported: deploy multiple proxies with load balancer. Customers run 10+ proxies for 1B+ events/day. No sampling required at any scale.

What is the compression ratio?

Typical: 85-95% compression from raw logs to Parquet+ZSTD. Syslog: 90% reduction. JSON: 85% reduction. NetFlow: 95% reduction. Actual ratio depends on data redundancy and schema complexity.

How fast are queries on historical data?

DuckDB: Scan 1TB Parquet in 30-60 seconds on laptop. ClickHouse: Sub-second queries on 100TB+ datasets. AWS Athena: 5-30 seconds for complex queries. Parquet columnar format enables blazing fast analytics.

What happens if proxy loses S3 connection?

Proxy buffers data to local disk (configurable limit: 10GB-1TB). Automatically resumes upload when connection restored. Zero data loss during network outages. Alerts sent if buffer exceeds threshold.

Can ByteFreezer handle schema changes?

Yes. Automatic schema evolution: new fields added without downtime. Schema versioning tracks changes over time. Backward compatible: old queries still work. Metadata catalog updated automatically.

What are the resource requirements?

Proxy: 2 CPU cores, 4GB RAM for 10K events/sec. Scales linearly. Storage: ~100GB/day for 100M events (after compression). Network: 100Mbps for 50K events/sec. Kubernetes: 3 nodes minimum for HA.
